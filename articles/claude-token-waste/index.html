Mechanics: how each improvement eliminates waste

Step through each optimization to see exactly what changed in the Claude API call structure.

Before — unbounded context
Every message re-sent on every Claude API call. After 20 turns across skills 1–7, the context going into skill 8 already contains 80,000+ tokens of irrelevant history.
← drag to simulate turn count →
→
After — capped context
Each skill gets a fresh Claude session. Maximum context = compressed skill file + 2 turns of its own history. No bleed from other skills. Context stays flat.
saves ~40% of total spend
Before — full SKILL.md every call
A 275-line skill file (~1,800 tokens) loaded in full on every turn. Verbose prose, long examples, edge-case notes — almost none needed for an edit pass.

Before — eval every skill every iteration
Claude Sonnet runs full eval loops regardless of what changed. Renaming a YAML field triggers the same eval as rewriting core logic. 800–1,200 tokens per eval run.
10 skills × 3 iterations × full eval:
30 eval runs · ~30,000 tokens
→
After — eval only on logic changes
Hash the logic section. If only description or formatting changed, skip eval entirely. Only trigger when step-by-step instructions were actually modified.
~40% of skills had logic changes needing eval:
4 eval runs · ~4,800 tokens
saves ~15% of total spend
